## 概述

## 1. Program-Aided LLM Reasoning（程序辅助推理）

### 代表工作
- **PAL** (Gao et al., 2023): LLM 生成 Python 代码来做算术
- **Program of Thoughts** (Chen et al., 2023): 类似 PAL 的思路

### 在确定性计算上的成功
PAL 在 GSM8K、符号数学等确定性计算任务上效果好——生成的代码大多正确。

### 在概率计算上的失败
但在需要实现非平凡概率算法（如变量消元）时崩溃：
- BLInD 深度 10: 代码执行成功率从 100% 降到 57%
- bnlearn ≥20 节点: 代码执行成功率 <10%

### 本文方法的三个区别

| 区别 | PAL | 本文 |
|------|-----|------|
| 合成粒度 | 每个实例一次 | 每个推理家族一次 |
| LLM 输出 | 可执行 Python | 类型化声明式 JSON |
| 推理时 LLM | 需要 | 完全不需要 |

## 2. Neuro-Symbolic Reasoning（神经符号推理）

### 逻辑/组合推理方向

| 工作 | 目标 | 形式化语言 |
|------|------|-----------|
| SatLM (Ye et al., 2023) | 布尔可满足性 | SAT |
| Logic-LM (Pan et al., 2023) | 一阶逻辑 | FOL |
| LINC (Olausson et al., 2023) | 定理证明 | FOL |
| Logic.py (Kesseli et al., 2025) | 逻辑推理 | 专用 DSL |
| CP (Michailidis et al., 2024) | 约束规划 | CP 规格 |

**关键区别**: 以上所有工作都处理**逻辑或组合**推理，没有一个处理**概率**情况。

### 概率推理方向

| 工作 | 目标 | 方法 |
|------|------|------|
| BLInD (Nafar et al., 2025) | BN 推断 | ProbLog 代码生成 |
| QUITE (Schrader et al., 2024) | 真实贝叶斯推理 | Benchmark 评估 |

### 本文与 BLInD 的三个区别

| 区别 | BLInD (ProbLog) | 本文 |
|------|----------------|------|
| 范围 | 仅 BN 推断 | 多个推理家族（同一组原语） |
| 粒度 | 每个实例翻译为 ProbLog | 每个家族编译一次 |
| 接口 | LLM 写 PPL 代码 | LLM 输出约束在 DSL 类型系统内（已知家族填 TaskSpec JSON，新家族组合核心运算） |

BLInD 和 QUITE 都是逐实例操作，且不区分 Parse/Compute/Decide 作为不同的失败模式。

## 3. Tool Use and Library Learning（工具使用与库学习）

### 代表工作
- **Toolformer** (Schick et al., 2023): LLM 自主调用外部工具
- **ReAct** (Yao et al., 2023): 推理 + 行动交替
- **LILO** (Grand et al., 2024): 通过迭代抽象学习可复用代码库
- **DreamCoder** (Ellis et al., 2021): 程序归纳 + 库学习

### 与本文的区别

| 维度 | 工具使用/库学习 | 本文 |
|------|---------------|------|
| LLM 角色 | 推理时调用工具 | 编译时归纳规格 |
| 输出 | 可执行代码 | 声明式规格 |
| 验证 | 隐式（看是否 work） | 显式三门验证 |
| 信任边界 | "代码碰巧能用？" | "规格通过验证？" |

本文特意比通用 PPL（WebPPL, Stan, Pyro）更窄：只做离散精确推断，LLM 的输出被约束在类型化的 DSL 原语范围内（已知家族填 TaskSpec JSON，新家族用 7 个核心运算组合），而非编写任意可执行代码。

---

## 4. LLM-Guided Program Synthesis（LLM 引导的程序合成）

### 代表工作
- **Ellis et al., 2023**: 贝叶斯概念学习——LLM 提出假设，按先验+似然重新加权
- **FunSearch** (Romera-Paredes et al., 2024): 搜索程序空间寻找数学发现
- **Li et al., 2024**: 评估 LLM 的 Programming-by-Example 能力

### 关键区别

Ellis 的贝叶斯概念学习仍然**在每个测试实例上依赖 LLM**（运行时贝叶斯过程）。本文的归纳器产出**形式化规格**，编译后完全移除 LLM。

---

## 5. Reasoning-Time Search and Autoformalization

### 推理时搜索
- **Tree-of-Thoughts** (Yao et al., 2024): 探索多个推理路径
- 相关方法保留 LLM 在推理循环中，每次查询都有成本，无正确性保证

### 自动形式化
- **Autoformalization** (Wu et al., 2022): 将非形式数学翻译为形式规格
- 本文归纳器做的事类似——但目标是窄的类型化 DSL，不是通用形式语言
- 验证由领域特定验证器完成，不是通用证明助手

## 本文的独特定位

综合以上对比，论文的定位可以概括为：

```
                    逻辑推理              概率推理
                    ┌──────────────┐     ┌──────────────┐
实例级              │ SatLM, Logic-LM│     │ BLInD (ProbLog)│
(每个问题一次)      │ LINC, Logic.py │     │ PAL, QUITE     │
                    └──────────────┘     └──────────────┘
                    ┌──────────────┐     ┌──────────────┐
家族级              │              │     │              │
(每个家族一次)      │ (未见先例)    │     │  ★ 本文 ★    │
                    └──────────────┘     └──────────────┘
```

**本文声称的新颖性不在于单个组件**（DSL、程序合成、验证器都有先例），而在于**诊断分解 + 编译一次的家族级求解器归纳的组合**。

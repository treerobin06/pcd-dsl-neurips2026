# 论文概述与核心思想

## 一句话总结

LLM 能理解概率问题的结构（Parse ~95%）但无法可靠执行概率计算（Compute 22%-78%），论文通过让 LLM 一次性归纳出声明式任务规格（TaskSpec），编译为确定性求解器，实现 100% 准确率且成本降低 14 倍。

---

## 核心观察

论文的出发点是一个反直觉的发现：

> **LLM 不是不懂概率推理，而是算不对。**

具体而言：
- **Parse（解析问题结构）**: 82%-100% 准确率 —— LLM 能正确提取变量、条件概率表、查询条件等
- **Compute（执行概率计算）**: 22%-78% 准确率 —— LLM 无法可靠地执行变量消元、后验更新等数学运算
- **Decide（基于结果决策）**: 100% 准确率 —— 给 LLM 正确答案，它总能做出正确决策

这个"理解但不会算"的不对称性在 6 个模型、3 个厂商、5 个推理家族上一致复现。

---

## 核心思想：编译一次，精确推理

既然 LLM 擅长"理解结构"但不擅长"执行计算"，论文提出**分工**：

1. **编译时（一次性）**: LLM 分析少量任务样本（最少 1 个即可，论文实验中用 3-5 个），归纳出任务的数学结构，不写通用代码
2. **推理时（无 LLM）**: 确定性编译器将 TaskSpec 编译为精确求解器，完全不依赖 LLM

这个思路的关键洞察是：
- LLM 的强项（语言理解、结构归纳）只需要执行**一次**
- LLM 的弱项（数值计算）被完全**移除**
- 推理时零 LLM 成本，结果完全确定

---

## 两大贡献

### 贡献 1：编译一次的求解器归纳系统

| 组件 | 功能 | 关键特性 |
|------|------|---------|
| **概率 DSL** | 7 个核心运算 + 3 个 Family Macro | 类型化，覆盖离散概率推理 |
| **确定性编译器** | TaskSpec → 精确求解器 | 相同输入 → 相同输出，0.0 最大误差 |
| **验证导向归纳器** | LLM 分析样本 → TaskSpec（已知家族）或核心运算组合（新家族） | GPT-4o-mini 即可，最少 1 个样本（实验中用 3-5 个） |
| **三门验证器** | 代码完整性 / 金标准匹配 / 参考匹配 | Gate 3 可选，不影响正确性 |

**核心结果**:
- GPT-4o-mini（最便宜模型）归纳的求解器 = GPT-5.4 自由代码生成的结果
- 成本降低 14 倍（$0.008 vs $0.11）
- 在真实网络（≥20 节点）上，PAL 完全失败（0-3%），DSL 保持 100%

### 贡献 2：诊断证据与边界分析

| 发现 | 证据 |
|------|------|
| 计算是主要瓶颈 | 6 个模型 × 5 个任务家族，Compute 始终最低 |
| 模型扩展无法弥合差距 | GPT-4o-mini→GPT-5.4（67× 价格），BN Compute 仅提升 9pp |
| 方法有明确边界 | DeLLMa 上表现不佳——因为瓶颈是分布估计而非计算 |

## 关键术语表

| 术语 | 含义 |
|------|------|
| **PCD** | Parse-Compute-Decide，三阶段诊断框架 |
| **DSL** | Domain-Specific Language，概率领域特定语言 |
| **TaskSpec** | 声明式 JSON 任务规格，LLM 归纳的输出 |
| **Gold-Injection** | 注入正确中间结果来隔离测试各阶段 |
| **Family Macro** | 常见推理模式的语法糖（如变量消元、softmax 偏好） |
| **Compile-once** | 每个推理家族只需编译一次，不是每个实例 |
| **PAL** | Program-Aided Language Models，LLM 每个实例生成代码 |
| **BLInD** | Bayesian network Language Inference Dataset |
| **VE** | Variable Elimination，变量消元算法 |
| **LOO** | Leave-One-Out，留一法泛化测试 |

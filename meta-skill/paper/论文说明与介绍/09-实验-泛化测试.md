# 实验：泛化测试（Held-Out 推理家族）

> 关联文档：[[04-方法-概率DSL与编译器]] | [[05-方法-TaskSpec与归纳器]] | [[12-数据集详解]]

---

## 实验目的

论文的 DSL 包含 3 个 Family Macro（softmax_pref, beta_bernoulli, ve_query），分别对应 3 个训练时的推理家族。一个自然的质疑是：

> "DSL 是否只是在训练家族上过拟合了？换一个新家族就不行？"

本实验用**两个 DSL 设计时完全未考虑的推理家族**来回答这个问题。

---

## 两个 Held-Out 推理家族

### Naive Bayes (NB) 医学诊断

- **任务**: 根据症状推断最可能的疾病
- **计算**: $p(d | \mathbf{s}) \propto p(d) \prod_j p(s_j | d)$
- **数据**: 合成生成，3-6 种疾病，4-8 个症状
- **测试规模**: 200 个问题实例
- **DSL 中有对应 Macro 吗**: **没有**

### HMM 前向滤波

- **任务**: 根据观测序列推断最后一个时间步的隐状态
- **计算**: $\alpha_t(s) \propto p(o_t | s) \sum_{s'} \alpha_{t-1}(s') p(s | s')$
- **数据**: 合成生成，3-5 个隐状态，4-8 个观测步
- **测试规模**: 100 个问题实例
- **DSL 中有对应 Macro 吗**: **没有**

### 为什么 HMM 特别有说服力

HMM 前向滤波在结构上与所有 3 个训练家族都不同：
- 它需要**时间轴上的顺序迭代推理**
- 每个时间步执行 multiply → marginalize → normalize
- 这是一种**训练 Macro 中完全不存在的计算模式**

如果归纳器能用核心运算组合出正确的 HMM 求解器，就说明它**不是在选择已有 Macro，而是在构造新的运算序列**。

## 实验设计

### 对比条件

| 条件 | 描述 |
|------|------|
| **Direct Answer** | LLM 直接回答（无代码、无工具） |
| **Compile-time Free Code** | LLM 看 3-5 个样本后写**不受约束的** Python 求解器（可用任意 Python 库和算法） |
| **Core-ops Constrained** | LLM 看 3-5 个样本后，用 DSL 的 7 个核心运算**组合出计算流程**（不使用 Macro，也不写任意 Python） |

> **注意**: "Core-ops Constrained" 和 "Compile-time Free Code" 都是编译时（一次性）的代码生成方法。区别在于前者将 LLM 的输出**约束在 7 个类型化 DSL 原语内**，后者允许写任意 Python。论文 Table 3 中两者均达到 100%，说明 DSL 原语的约束没有损失表达能力。
>
> 这与训练家族上的 TaskSpec 归纳流程不同——训练家族有预定义的 `inference_family` enum 值（如 `variable_elimination`），编译器可直接匹配 Macro。对于 NB/HMM 这类没有对应 enum 值的新家族，归纳器需要**自行组合核心运算序列**来构造求解器，这本质上是受约束的程序合成。

### 同时进行 PCD 诊断

在这两个新家族上也进行 PCD 诊断，验证"计算瓶颈"的模式是否复现。

---

## 结果

### 任务准确率

| 条件 | NB (GPT-4o-mini) | NB (GPT-5.4) | HMM (GPT-4o-mini) | HMM (GPT-5.4) |
|------|-----------------|-------------|-------------------|---------------|
| Direct Answer | 44.0% | 68.5% | 32.0% | 61.0% |
| Compile-time Free Code | **100.0%** | **100.0%** | **100.0%** | **100.0%** |
| Core-ops Constrained | **100.0%** | **100.0%** | **100.0%** | **100.0%** |

**核心发现**:
- 仅用核心运算（无 Macro）在两个 held-out 家族上都达到 100%
- 与自由代码生成的结果完全一致
- **确认 Macro 是纯粹的语法糖**

### PCD 诊断

| 阶段 | NB (mini) | NB (5.4) | HMM (mini) | HMM (5.4) |
|------|-----------|----------|------------|----------|
| Parse | 3.0% [1,8] | 100.0% [98,100] | 100.0% | 99.0% [95,100] |
| Compute\|GoldParse | 37.0% [30,44] | 64.5% [58,71] | 27.0% [19,37] | 53.0% [43,63] |
| Decide\|GoldPosterior | 100.0% [98,100] | 100.0% [98,100] | 98.0% [93,100] | 100.0% |

**计算瓶颈的诊断签名在新家族上完美复现**: 低 Compute、高 Decide。

## 异常情况说明

论文诚实报告了两个异常：

### 异常 1: GPT-4o-mini 在 NB 上 Parse 仅 3%

- **原因**: GPT-4o-mini 倾向于省略"未出现症状"的条目（absent-symptom entries）
- **影响**: 仅影响 Parse 测试，不影响 core-ops 求解器结果（因为 core-ops 不依赖 LLM 的 Parse）
- **解读**: 这反而说明了 DSL 方法的优势——即使 LLM 在实例级 Parse 上表现差，家族级归纳仍然成功

### 异常 2: GPT-4o-mini 在 HMM 上 Decide 降到 98%

- **原因**: 部分 HMM 实例的后验分布非常接近（near-tied posteriors），argmax 在浮点精度下不稳定
- **影响**: 仅 2 个实例受影响
- **解读**: 这是决策面极端平坦时的 edge case，不影响核心结论

## 此实验解答的核心问题

| 问题 | 答案 | 证据 |
|------|------|------|
| Macro 是必需的吗？ | 不是，核心运算足够 | NB + HMM 均 100%，无需 Macro（Core-ops Constrained = 100%） |
| DSL 能泛化到新家族吗？ | 是 | 两个未见过的家族上 100% |
| PCD 模式是普遍的吗？ | 是 | 新家族上也是低 Compute、高 Decide |
| 归纳器只是选 Macro 吗？ | 不是，它能构造新序列 | HMM 的 multiply-marginalize-normalize 循环 |

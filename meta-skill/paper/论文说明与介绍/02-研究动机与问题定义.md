## 研究动机

### 1. LLM 在概率推理上的系统性失败

论文开篇给出了一个具有冲击力的数据：

> 让前沿 LLM 对一个 10 节点贝叶斯网络做变量消元，**89% 的时间会算错**。
> 但让它**提取变量、解析条件概率表、确定查询**，**99%** 的时间能成功。

这个差距不是某个模型的个案，而是跨模型、跨任务的系统性现象：
- 偏好学习任务：LLM 比简单贝叶斯 Oracle 低 20-60 个百分点
- BN 推断任务：深度增加时准确率从 ~82% 暴跌到个位数
- 多臂赌博机任务：同样表现远低于贝叶斯基准

### 2. 现有解决方案的碎片化

针对 LLM 概率推理失败，现有方案都是**针对特定任务的临时修补**：

| 方案 | 适用范围 | 局限 |
|------|---------|------|
| ProbLog 代码生成 | BN 推断 | 仅限贝叶斯网络，逐实例翻译 |
| 微调贝叶斯预测 | 偏好学习 | 需要大量标注数据，不可迁移 |
| CoT 提示 | 通用 | 在概率任务上甚至可能降低性能 |

每个新任务家族都需要从头设计新的工具，缺乏**统一的解决框架**。

### 3. 实际应用场景

论文举了一个具体的应用场景来说明为什么这个问题重要：

**交互式偏好学习**（如购物、旅行推荐）：
- LLM 助手观察用户的一系列选择
- 需要推断用户的隐藏偏好模型（贝叶斯后验更新）
- 然后推荐下一个选项

这种交互需要**反复的后验更新**——正是 LLM 无法可靠执行的概率计算。

## 问题的形式化定义

### 通用概率推理骨架

论文将所有测试任务抽象为同一个骨架：

$$
\text{观察数据 } \mathbf{x} \rightarrow \text{计算后验 } p(\theta | \mathbf{x}) \rightarrow \text{做最优决策 } a^* = \arg\max_a \mathbb{E}_{p(\theta|\mathbf{x})}[u(a, \theta)]
$$

不同的是**推理家族**——即具体的计算方式：

| 推理家族     | 计算方式                            | 对应任务                   |
| -------- | ------------------------------- | ---------------------- |
| 假设枚举     | 遍历所有可能的参数 $\theta$，softmax 似然更新 | 偏好学习（Flight/Hotel）     |
| 共轭更新     | Beta 分布的闭式更新                    | 多臂赌博机（TextBandit）      |
| 变量消元     | 因子乘积 + 边际化                      | BN 推断（BLInD + bnlearn） |
| 朴素贝叶斯    | 似然独立假设下的后验                      | 医学诊断（held-out）         |
| HMM 前向滤波 | 时间轴上的迭代乘-边际化-归一化                | 隐马尔可夫模型（held-out）      |

### 编译一次的目标

论文将"编译一次"形式化为以下优化目标：

给定 $k$ 个标注样本 $\mathcal{D}_F^{(k)} = \{(x_i, y_i^*)\}_{i=1}^k$：

1. **归纳器** $I_\phi$（LLM）推断任务规格 $\hat{\tau}$
2. **编译器** $C$ 将规格编译为求解器 $\hat{S} = C(\hat{\tau})$
3. 目标：$\mathbb{E}_{x \sim p_F}[\ell(\hat{S}(x), y^*(x))] = 0$

关键约束：
- 归纳器只在**每个推理家族**调用一次（不是每个实例）
- 推理时 $\hat{S}$ 完全不依赖 LLM
- 输出是确定性的（相同输入 → 相同输出）

## 与已有工作的关键区别

### 区别 1：家族级 vs 实例级

| 对比维度 | PAL / ProbLog | 本文方法 |
|---------|---------------|---------|
| 粒度 | 每个问题实例生成一段代码 | 每个推理家族归纳一次 |
| LLM 调用次数 | N 次（N = 问题数） | 1 次 |
| 推理时是否需要 LLM | 是 | 否 |

### 区别 2：约束输出 vs 自由代码

| 对比维度 | PAL / 自由代码生成 | 本文方法 |
|---------|-------------------|---------|
| LLM 输出 | 任意 Python/ProbLog 代码 | 已知家族：声明式 JSON（TaskSpec 填字段）；新家族：用 7 个核心运算组合计算流程 |
| 错误类型 | 语法错误 + 逻辑错误 + 算法错误 | 已知家族：仅字段填写错误；新家族：运算组合错误（但搜索空间远小于任意代码） |
| 最低模型要求 | GPT-5.4 才能写出正确 VE | GPT-4o-mini 即可（因为输出空间被类型系统约束） |

### 区别 3：已有诊断的空白

论文指出，之前的工作记录了 LLM 在概率推理上的失败，但**没有区分**失败是发生在：
- 理解问题结构时（Parse）
- 执行数学计算时（Compute）
- 根据结果决策时（Decide）

PCD 框架正是为了填补这个诊断空白而设计的。

## 核心假设链

论文的论证逻辑遵循以下假设链：

```
假设 1: LLM 的概率推理失败主要在计算环节
   ↓ [PCD 诊断验证]
假设 2: 既然 LLM 能理解结构，它应该也能做"家族级归纳"
   ↓ [Inductor 实验验证]
假设 3: 归纳出的 TaskSpec 经确定性编译后能正确求解
   ↓ [等价性验证 + 1800+ 测试实例]
假设 4: 这个方法能泛化到 DSL 设计时未考虑的推理家族
   ↓ [NB + HMM held-out 验证]
```

每个假设都有对应的实验证据支撑，详见 [[07-实验-Baseline对比]] 至 [[11-分析-适用边界与模型扩展]]。

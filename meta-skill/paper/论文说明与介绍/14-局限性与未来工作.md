论文在 Limitations 部分（Section 8）坦诚讨论了方法的边界。这是一个有价值的部分，对审计者了解方法的实际适用范围至关重要。

## 一、方法范围与扩展性

### DSL 的覆盖范围

**当前支持**:
- 离散变量的精确推断
- 5 个推理家族（3 个训练 + 2 个 held-out）

**不支持**:
- 连续变量
- 近似推断方法（MCMC, 变分推断等）
- 预测瓶颈的任务（如 DeLLMa，见 [[11-分析-适用边界与模型扩展]]）

**扩展方向**: 论文指出扩展到连续分布和近似后端是最自然的下一步。

### 推理家族的广度

- 5 个推理家族 × 8 个数据集建立了 PCD 模式
- Held-out 泛化在 NB (n=200) 和 HMM (n=100) 上验证
- 论文建议在 QUITE 等外部 benchmark 上进一步验证

### DSL 运算的完备性

- 当前 7 个核心运算 + 3 个 Macro 是作者基于概率推理领域知识设计的
- **扩展到新推理家族可能需要添加新运算**：
  - `log`: 用于信息论量（如 KL 散度）
  - `max-marginalize`: 用于 MAP 推断
- 论文**不声称覆盖所有离散推断任务**

## 二、PCD 方法论

### 不是因果识别

PCD 使用**干预性探针**（injection interventions），不是严格的因果推断。它展示了"给定正确的中间结果时各阶段的表现差异"，但不构成因果关系的形式化证明。

### Compute|GoldParse 的混淆

Compute|GoldParse 阶段测试的是"给定完美结构输入，LLM 能否计算正确"。但这里混淆了两件事：
1. **算法选择**: LLM 是否选择了正确的算法（如变量消元 vs 错误的方法）
2. **算术执行**: LLM 的算术运算是否正确

PCD 无法区分这两种失败模式。

### Decide 分数可能偏高

Decide 100% 看起来很惊人，但论文指出：
- 这里的决策面很简单：对期望效用取 argmax
- 这不一定代表 LLM 在**复杂决策**（多目标、约束优化等）上也能保持完美

### BN 深度与 token 长度的混淆

更深的 BN 意味着更长的 token 输入。Compute 的退化可能部分是因为输入太长。但反驳：Parse 在所有深度上保持 ≥96%，暗示 token 长度本身不是主因。论文建议通过控制实验来分离这个混淆。

## 三、归纳器

### 当前成功可能反映任务简单性

归纳器在所有测试任务上 100% 成功（6/6 LOO，第一轮通过）。但论文承认：
- 这可能是因为测试的任务足够"干净"
- 在更大的状态空间和对抗性规格上需要进一步压力测试
- 归纳器是随机 LLM 调用——初步可靠性测试（20 次 × 2 家族）显示 100%，但需要更广泛的测试

### PCD 与家族级归纳的关系

PCD 展示的是 LLM 能做**实例级** Parse（单个问题的结构提取）。论文声称 LLM 也能做**家族级**归纳（从几个样本推断整个家族的数学结构），但这是一个更强的能力。

论文在这里的立场：
> "PCD demonstrates that LLMs can parse individual problem instances; family-level induction is a stronger capability that our experiments validate empirically rather than derive from PCD alone."

## 四、验证与正确性

### 经验验证 vs 形式化证明

- **"0.0 最大误差"** 是通过与 gold solver 交叉验证得到的，不是形式化正确性证明
- Gold solver 本身是独立实现的，并在 BN 上与 pgmpy 交叉验证
- 但仍然是经验验证，存在未覆盖的 edge case 的理论可能

### Gate 3 是可选的

- Gate 3 需要 gold reference solver，这在新任务上可能不可用
- Gate 1+2 仅需要小量 held-out 验证样本，实际可部署
- 消融实验确认 Gate 1+2 即可保证正确性

## 五、统计报告

### 确定性 vs 随机

- DSL 求解器输出是**确定性的**（没有采样方差），所以不需要跑多次
- PCD 实验报告了 95% CI（Tables 1, 3, Figure 2）
- TextBandit 只有 4 种配置，统计力度有限

### 可复现性

- 所有 prompt、规格、编译求解器将在论文接受后发布
- 实验使用商业 API（OpenAI, Anthropic, Google），可通过相同 API 复现

---

## 六、论文自己承认的"这不是局限"

论文有一段有趣的声明：

> "We note that the system's accuracy on test instances is the solver's achievement by design; the LLM's contribution is the one-time structural induction that makes the solver possible, a division of labor we view as a feature rather than a limitation."

即：测试准确率是求解器的成就（不是 LLM 的），这是**设计意图**，不是局限。LLM 的贡献是一次性的结构归纳——正是这种分工使系统有效。

## 对审计者的总结

| 局限类别 | 严重程度 | 论文是否承认 | 审计建议 |
|---------|---------|-------------|---------|
| 仅限离散精确推断 | 中 | ✓ | 确认声明范围 |
| PCD 非因果识别 | 低 | ✓ | 这是常见的方法论限制 |
| Compute 混淆算法选择与算术 | 中 | ✓ | 可以要求分离实验 |
| 归纳器成功可能因任务简单 | 中 | ✓ | 需要在更难的任务上测试 |
| 等价性是经验验证非形式化 | 低 | ✓ | 学术论文常见做法 |
| DeLLMa 数据量小 | 低 | 部分 | 20 个问题统计力度有限 |
| BN 深度-token 混淆 | 低 | ✓ | Parse 保持高准确率提供反驳 |

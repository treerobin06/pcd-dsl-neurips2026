# 方法：三门验证器

> 关联文档：[[05-方法-TaskSpec与归纳器]] | [[04-方法-概率DSL与编译器]] | [[10-分析-策略消融与端到端验证]]

---

## 目的

三门验证器（Three-Gate Verifier）是在编译器输出求解器后，**部署前**的质量检查机制。它确保编译出的求解器是正确的，如果不正确，则提供诊断信息帮助归纳器修正。

---

## 三门设计

### Gate 1: Code Sanity（代码完整性检查）

- **检查内容**: 编译后的求解器是否能正常执行，不抛出异常
- **输入**: 编译后的求解器 + 任意测试输入
- **判定**: 执行无错误 → Pass / 抛出异常 → Fail
- **失败时反馈**: 异常类型和错误信息，供归纳器修正 TaskSpec

这是最基本的检查——求解器至少要能运行。

### Gate 2: Ground Truth（金标准匹配）

- **检查内容**: 求解器在 held-out 验证样本上的输出是否与已知正确答案一致
- **输入**: 编译后的求解器 + 验证集 $\mathcal{D}_{\text{val}}$（与归纳样本和最终测试集**完全不相交**）
- **判定**: 所有验证样本输出匹配（$\|\hat{S}(x) - y\|_\infty < \epsilon$）→ Pass
- **失败时反馈**: 具体哪些样本不匹配、预期值 vs 实际值

> **关键设计**: 验证集与归纳样本不重叠（disjoint），防止过拟合归纳样本。

### Gate 3: Reference Match（参考求解器匹配）

- **检查内容**: 编译后的求解器与独立实现的 gold reference solver 输出是否一致
- **输入**: 编译后的求解器 + gold reference solver + 测试集
- **判定**: 所有测试实例上两者输出一致 → Pass
- **状态**: **可选的**，不是部署的必要条件

---

## 形式化定义（Eq. 5）

$$V(\hat{S}, \mathcal{D}_{\text{val}}) = \underbrace{\mathbf{1}\{\hat{S} \text{ executes}\}}_{G_1: \text{sanity}} \wedge \underbrace{\mathbf{1}\{\forall (x,y) \in \mathcal{D}_{\text{val}}: \|\hat{S}(x) - y\|_\infty < \epsilon\}}_{G_2: \text{ground truth}} \wedge \underbrace{\mathbf{1}\{\hat{S} \equiv S^*\}}_{G_3: \text{reference (opt.)}}$$

验证的逻辑是**串行**的：
1. 先过 Gate 1（不能运行的求解器没必要继续测试）
2. 再过 Gate 2（通过基本正确性检查）
3. 最后 Gate 3（可选的额外安全检查）

---

## Gate 3 消融实验

论文通过消融实验证明 Gate 3 **不是必需的**：

| 数据集 | 仅 Gate 1+2 | Gate 1+2+3 | Gold Match |
|--------|------------|-----------|-----------|
| Hotel | Pass | Pass | 100% |
| Flight-2F | Pass | Pass | 100% |
| Flight-3F | Pass | Pass | 100% |
| Flight-5F | Pass | Pass | 100% |
| Flight-6F | Pass | Pass | 100% |
| BLInD depth-OOD | Pass | Pass | 100% |

6 个 LOO 数据集仅用 Gate 1+2 就全部通过，且与 gold solver 100% 一致。

**Gate 3 不必需的意义**:
- 实际部署时**不需要预先有一个完整的 gold solver**
- 只需要少量验证样本（Gate 2 的输入）即可保证正确性
- 消除了"先有鸡还是先有蛋"的问题——如果已经有 gold solver，还需要 DSL 吗？

## 验证失败时的修正流程

```
归纳器生成 TaskSpec (Round 0)
     │
     ▼
编译器编译 → 求解器
     │
     ▼
Gate 1: 执行检查
     │ Fail → 诊断信息: "KeyError in line 42: 'softmax_choice' not recognized"
     │        → 归纳器修正 TaskSpec (Round 1)
     │ Pass ↓
Gate 2: 金标准检查
     │ Fail → 诊断信息: "Sample #3: expected 0.72, got 0.45"
     │        → 归纳器修正 TaskSpec (Round 2)
     │ Pass ↓
Gate 3: 参考匹配 (可选)
     │ Fail → 诊断信息: "Solver disagrees with gold on 12/250 instances"
     │ Pass ↓
验证通过 → 部署求解器
```

最多允许 $R_{\max} = 3$ 轮修正。如果 3 轮后仍未通过，报告失败。

但在实际实验中，**所有测试任务都在第一轮通过**（见 [[05-方法-TaskSpec与归纳器]] 中的可靠性数据）。

## 完整算法（Algorithm 1）

```
输入: 归纳样本 D_train, 验证样本 D_val (不相交)
输出: 验证通过的求解器 S 或失败

1. spec ← LLM_induce(D_train)          // GPT-4o-mini 归纳 TaskSpec
2. FOR r = 1 TO R_max:
3.   S ← Compile(spec)                  // 确定性编译
4.   g1 ← Gate1_Sanity(S)               // 能执行？
5.   IF NOT g1:
6.     spec ← LLM_refine(spec, diagnostics)
7.     CONTINUE
8.   g2 ← Gate2_GroundTruth(S, D_val)   // 输出正确？
9.   IF g1 AND g2:
10.    g3 ← Gate3_Reference(S)           // 可选: 匹配 gold?
11.    RETURN S                          // 部署验证通过的求解器
12.  spec ← LLM_refine(spec, diagnostics)
13. RETURN FAILURE
```

## 信任边界的转移

论文强调了验证器带来的一个重要的信任模型变化：

| 方面 | PAL / 自由代码生成 | 本文方法 |
|------|-------------------|---------|
| 信任问题 | "LLM 生成的代码碰巧能工作吗？" | "声明式规格通过了经验验证吗？" |
| 信任对象 | LLM 的代码生成能力 | 编译器 + 验证器 |
| 失败模式 | 运行时才发现 bug | 部署前就被 Gate 拦截 |

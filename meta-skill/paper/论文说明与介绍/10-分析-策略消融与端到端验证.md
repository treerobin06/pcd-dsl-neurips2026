## 一、23 策略消融实验

### 实验目的

从另一个角度补充 PCD 的发现：通过系统地变化**从 LLM 卸载的计算量**，展示准确率如何随卸载量单调递增。

### 实验设置

- **数据集**: Flight 偏好学习，624 个样本
- **模型**: GPT-4o-mini
- **策略数**: 23 种（原论文系统中已有的 18 种 + 额外分组）

### 策略分类与结果

策略按"卸载的计算量"从少到多排列，形成一个**单调阶梯**：

#### 层级 1: 无计算卸载（33-37%）

| 策略 | 准确率 | 95% CI | 说明 |
|------|--------|--------|------|
| baseline（无注入） | 36.8% | [33.1, 40.7] | LLM 独立推理 |
| random_rec | 33.5% | [31.8, 35.1] | 随机推荐控制组 |

#### 层级 2: 纯 CoT（≤33%，低于随机基线）

| 策略 | 准确率 | 说明 |
|------|--------|------|
| cot_reflect | 32.2-33.1% | 要求 LLM 先分析偏好模式 |
| cot_bayesian | 同上 | 教 LLM 贝叶斯步骤（不给结果） |
| few_shot | 同上 | 给 2 个推理范例 |
| cot_self_consistency | 同上 | 5 路并发 + 多数投票 |

**关键发现**: 纯 CoT 完全无效，4 种变体全部≤随机基线。LLM 无法通过"更认真地思考"来改善概率计算。

#### 层级 3: Prompt 注入（35-50%）

| 策略 | 准确率 | 95% CI | 注入内容 |
|------|--------|--------|---------|
| direct_rec | 34.8% | [33.1, 36.5] | 仅推荐 |
| nl_pref_rec | 38.8% | [37.0, 40.5] | 偏好描述 + 推荐 |
| full_math | 49.9% | [46.1, 53.7] | 完整数学分析 |

#### 层级 4: Function Calling（37-58%）

| 策略 | 准确率 | 95% CI | 通道增益 |
|------|--------|--------|---------|
| tool_empty | 37.1% | [35.4, 38.8] | — |
| tool_random_rec | 36.3% | [34.6, 38.1] | +2.8pp |
| tool_nl_pref | 41.8% | [40.2, 43.6] | +9.4pp |
| tool_direct_rec | 53.5% | [51.7, 55.2] | +18.7pp |
| tool_nl_pref_rec | 56.8% | [54.9, 58.6] | +18.0pp |
| tool_use (full math) | **58.3%** | [52.1, 60.0] | +8.4pp |

#### 层级 5: Oracle（74.8%）

直接注入 gold 后验 → LLM 100% 服从。

### 内容 × 通道 2D 矩阵

| 注入内容 | Prompt | Function Calling | 通道增益 |
|---------|--------|-----------------|---------|
| 随机推荐 | 33.5% | 36.3% | +2.8pp |
| 直接推荐 | 34.8% | **53.5%** | **+18.7pp** |
| NL 偏好 | 32.5% | 41.8% | +9.4pp |
| NL 偏好+推荐 | 38.8% | **56.8%** | **+18.0pp** |
| 完整数学 | 49.9% | **58.3%** | +8.4pp |

**关键发现**:
1. 通道增益对**中等复杂度内容**最大（直接推荐: +18.7pp）
2. 对**无用内容**几乎无增益（随机: +2.8pp）
3. 对**已经高度信息化的内容**增益有限（完整数学: +8.4pp）
4. Function calling 主要提升的是**服从率**（LLM 更愿意采纳工具返回的建议）

### 与 PCD 的关系

23 策略消融与 PCD 互为补充：
- **PCD 说的是**: "LLM 不会算"（通过隔离各阶段）
- **策略消融说的是**: "帮 LLM 算得越多，结果越好"（通过变化卸载量）

两者指向同一个结论：**计算是瓶颈，卸载计算是解决方案**。

## 二、端到端 Pipeline 验证

### 实验目的

PCD 实验是**分阶段隔离测试**的。一个自然的问题是：

> "把各阶段串联起来，整个 pipeline 真的能 work 吗？"

### 实验设计

完整的端到端流程：
```
自然语言航班描述
  → GPT-4o-mini Parse（提取数值特征）
  → 确定性归一化
  → 编译好的 PreferenceSolver
  → 推荐结果
```

**输入示例**:
```
departure time: 02:00 PM, duration: 4 hr 24 min, price: $730
```

**LLM 需要做的**: 将 "02:00 PM" → 14.0，"4 hr 24 min" → 4.4，"$730" → 730 等

### 结果

| 指标 | 数值 |
|------|------|
| E2E 准确率 | **74.3%** [70.9%, 77.8%] |
| Gold-feature Oracle | 74.4% |
| 特征提取准确率 | 99.9%-100%（每个特征） |
| 与 Gold Solver 一致率 | 99.8% |
| 剩余 0.2% 差距来源 | 2 个 JSON 解析失败（不是提取错误） |

### 关键发现

1. **E2E 74.3% ≈ Gold Oracle 74.4%**: Pipeline 几乎完美复现了 gold-feature oracle
2. **特征提取近乎完美**: LLM 从自然语言中提取数值特征的能力极强
3. **与最佳 prompt 策略对比**: E2E 74.3% vs tool_use 58.3% = **+16pp**

E2E 没有达到 100% 是因为贝叶斯 Oracle 本身在 5 轮交互后只有 74.8% 准确率（偏好学习固有的不确定性）。Pipeline 的 74.3% 已经几乎等于理论上限。

## 三、Gate 3 消融

### 实验目的

验证三门验证器中的 Gate 3（Reference Match）是否是部署的必要条件。

### 实验设计

在 6 个 LOO 数据集上，关闭 Gate 3，仅使用 Gate 1 + Gate 2 进行验证。

### 结果

| 数据集 | 仅 Gate 1+2 | 与 Gold Solver 一致率 |
|--------|------------|---------------------|
| Hotel | Pass | 100% |
| Flight-2F | Pass | 100% |
| Flight-3F | Pass | 100% |
| Flight-5F | Pass | 100% |
| Flight-6F | Pass | 100% |
| BLInD depth-OOD | Pass | 100% |

**结论**: Gate 3 不是必需的。仅 Gate 1+2 就能保证正确性。

**这意味着**: 部署时不需要预先有一个 gold reference solver——只需要少量验证样本即可。
